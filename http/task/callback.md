# 异步回调推送

推荐使用回调获取打印结果，比轮询 result 更实时。

## 启用方式

登录快麦管理后台，进入**打印回调配置**页面，填写接收回调的 URL（**公网可达，建议 HTTPS**）。前提：该账号必须已绑定对应打印机。

## 触发时机

打印任务到达终态（成功 / 失败 / 异常）时，云平台向对接方回调地址发起 HTTP POST 请求。

## 回调请求体

`Content-Type: application/json`

| 参数名 | 类型 | 说明 |
|--------|------|------|
| jobId | String | 打印任务 id，与打印接口返回的 jobIds 元素一致 |
| sn | String | 打印机序列号 |
| code | Number | 任务结果码（3000=下发成功 / 2000=打印成功 / 2007=打印失败 / 2004=打印异常） |
| desc | String | 结果描述，例如"打印成功" |

```json
{
  "desc": "打印成功",
  "jobId": "1612514721477",
  "code": 2000,
  "sn": "KM118MW"
}
```

## 接入方实现要求

::: warning 必须返回
收到回调后，接入方必须以 **HTTP 200** 返回 `{"data":"OK"}`。云平台据此判定接收成功。
:::

```json
{"data":"OK"}
```

## 重试与幂等

- **重试**：未收到 HTTP 200，或响应体不是 `{"data":"OK"}`，云平台按策略**重推 2 次**（单条回调最多推送 3 次）。3 次都失败后不再推送，请配合 [result](/zh/http/task/print-result) 接口兜底。
- **幂等**：同一 jobId 可能被推送多次，业务侧务必用 jobId 做唯一键去重。
- **超时**：建议接入方处理 ≤ 3 秒。复杂业务请异步处理后**立即返回** `{"data":"OK"}`，避免触发重推。

## 与轮询 result 的对比

| 维度 | 异步回调 | 轮询 result |
|------|----------|-------------|
| 实时性 | 高（终态触发） | 取决于轮询间隔 |
| 调用频次 | 由平台主动推送 | 单 sn ≤ 2 次/秒；单次 jobId ≤ 50 |
| 可靠性 | 失败重推 2 次（最多 3 次） | 自行控制 |
| 适合场景 | 业务侧具备公网回调地址 | 内网或临时调试 |

::: tip 推荐方案
"回调为主、轮询兜底"的双保险方案：以回调为主链路，业务侧定时（低频）扫描未收到回调的 jobId，再调用 result 兜底。
:::
